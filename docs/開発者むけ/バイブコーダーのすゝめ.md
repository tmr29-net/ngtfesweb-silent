# AIエージェント開発：完全実践ガイド (作った本人はまだできていない模様)

## 1. 開発環境のセットアップ（初動）

AIが「自律的」に動くための土台を整えます。

* **`.ai-rules.md` の作成**
プロジェクトルートに配置し、AIに「常にこれを守れ」と命じます。
> **内容例:** 「技術スタックは Next.js / Tailwind。共通コンポーネントは `src/components/ui` を最優先で使い、DRY原則を徹底すること。テストがないコードは受け付けない。」


* **ディレクトリ構造の固定**
`tree -L 2` などのコマンド結果をREADMEに貼り付け、AIが「どこに何があるか」を瞬時に理解できるようにします。

## 2. デイリー・開発ワークフロー

複数人で開発する場合、以下のステップを1サイクル（1〜2時間単位）として回します。

### ステップ1：コンテキストの同期 (Human)

作業開始前に、他のメンバーの変更を取り込み、AIに現在の状態を教えます。

1. `git pull origin main` で最新化。
2. エージェントに対し：**「現在のリポジトリの構造と、最新の `package.json`（または依存定義）を確認して報告してください」**と指示。

### ステップ2：アトミック・プロンプティング (Human → AI)

一度に多くを頼まず、**「1つの目的」**に絞ります。

* **NG:** 「ログイン機能と、ついでにダッシュボードのUIも直して」
* **OK:** 「`auth/login.ts` に、メールアドレスの形式チェックを行うバリデーション関数を追加してください。既存の `utils/validate.ts` に似た関数がないか先に確認すること。」

### ステップ3：テストによるガードレール (AI)

AIがコードを書く前に、以下を徹底させます。

1. **「まず、期待する動作をするテストコードを `tests/` に作成してください」**
2. **「そのテストが失敗することを確認してから、実装に移ってください」**
3. **「実装後、テストがパスしたログを提示してください」**

### ステップ4：人間による「差分（diff）」レビュー (Human)

AIが勝手にファイルを消したり、無関係な箇所を書き換えたりしていないか確認します。

* `git diff` を実行し、意図しない変更があれば**「この箇所の変更は不要です。元に戻してください」**と即座に修正させます。

## 3. バグ修正・仕様変更の「型」

行き当たりばったりの修正を防ぐための具体的な手順です。

| 状況 | AIへの具体的な指示（テンプレート） |
| --- | --- |
| **バグ発生** | 「[エラー内容] が発生しました。まず `grep` でエラー箇所を特定し、再現テストを作成してください。修正案を出す前に原因を3行で説明してください。」 |
| **仕様変更** | 「[変更内容] を行います。修正が必要なファイルと、それによって影響を受ける既存機能（デグレードのリスク）をリストアップしてください。」 |
| **リファクタリング** | 「このディレクトリ内でDRY原則に反している（重複している）コードを3つ挙げてください。それらを共通化する案を出してください。」 |

---

## 4. 複数人開発での「衝突回避」ルール

* **ブランチ名のルール:** `ai/[担当者名]/[機能名]` （例: `ai/sato/fix-header`）
* **進捗共有:** 30分に一度、AIに現在の状況を `STATUS.md` に書き出させ、チームのSlack等に共有する。
* **「3回ルール」:** AIが同じエラーで3回解決に失敗したら、**人間がそのセッションを強制終了し、手動で修正するかヒントを与えてから新しいセッションを開始する。**（トークンの無駄遣いとコードの汚染を防ぐため）

---

## 5. セッション終了時の儀式

AIとのセッションを閉じるときは、必ず次への「バトン」を作成させます。

* **「ここまでの作業内容と、残っている課題、次に着手すべきステップを `NEXT_STEP.md` に簡潔にまとめてください。」**
* これにより、翌日や他のメンバーが作業を引き継ぐ際のスピードが劇的に上がります。
